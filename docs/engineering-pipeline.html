FEATURE Track

New capability · domain change · new bounded context — Full 7-phase pipeline

Phase 0 Intake
Phase 1 Discovery
Phase 2 Implementation
Phase 3 Review
Phase 4 QA
Phase 5 Docs
Phase 6 Deploy
Phase 0 — Intake & Track Selection
Classify the Request
engineering-manager
  • Classify into FEATURE / HOTFIX / CHORE / SPIKE / UI-ONLY
  • Identify affected bounded contexts
  • Assign severity and target milestone
  • Check for conflicts with in-flight work
GATE: Track confirmed before anything else starts
⟵ PARALLEL ⟶
Phase 1A — Domain Analysis
Rules, Criteria & Edge Cases
personal-finance-analyst
  • Domain rules (Money, soft-delete, @Transactional)
  • Given/When/Then acceptance criteria
  • Financial correctness constraints
  • Edge cases catalog
Phase 1B — Architecture Brief
Ports, Adapters & Implementation Order
tech-lead
  • New inbound/outbound port interfaces
  • Domain service signatures (pure Java)
  • JPA entity & Liquibase migration plan
  • N+1 risks, complexity estimate (S/M/L)
  • Implementation dependency order
  • ADR authored or referenced
Phase 1C — Convergence
Unified Feature Brief
engineering-manager
  • Reconciles analyst + tech-lead outputs — conflicts resolved here, not in Phase 2
  • Out-of-scope explicitly listed
  • Security & performance considerations captured
Phase 1D — Stakeholder Gate
Feature Brief Review & Approval
product-owner
  • Acceptance criteria correct and complete?
  • Scope right — nothing missing, nothing extra?
  • Complexity estimate acceptable for milestone?
  • Security / compliance concerns flagged?
GATE: Feature Brief approved — implementation cannot start until this passes
Phase 2 — Pre-Implementation NEW
Branch & PR Strategy
devops-engineertech-lead
  • Create feature branch from main (naming: feature/TICKET-description)
  • Define PR merge rules (squash vs merge, required reviewers)
  • Set up draft PR early so work is visible to the team
  • Confirm CI pipeline will run on push to feature branch
⟵ PARALLEL ⟶
Phase 2A — Backend
Domain → Adapters → REST
full-stack-dev
  • Domain model (pure Java — zero Spring/JPA)
  • Port interfaces (UseCase / Query / PersistencePort)
  • Domain service + Config @Bean wiring
  • JpaEntity, JpaMapper, Liquibase migration
  • REST controller + record DTOs
  • Cross-context ACL adapters if needed
GATE: ./gradlew test passes · ArchUnit clean · migrations run on fresh schema
Phase 2B — UX / Frontend Design
Components & Design Spec
ux-ui-designer
  • Component hierarchy for new screens
  • Mobile-first breakpoints (375px)
  • Loading, empty, and error states specified
  • ARIA roles & keyboard navigation spec
  • Design system tokens (shadcn/ui + Tailwind)
GATE: Design approved · mobile layouts specified
API contract stable
Phase 2C — Frontend Implementation
React + TypeScript Components
full-stack-dev
  • React components per design spec
  • TypeScript types aligned with backend DTOs
  • API client functions (Authorization: Bearer token)
  • Form validation matching backend rules
  • No any types in new code
GATE: npm run build clean · no TypeScript errors · mobile renders correctly
⟵ PARALLEL REVIEW STREAMS ⟶
Phase 3A — Architecture Review
Domain Zone & ADR Compliance
tech-lead
  • No Spring/JPA/Lombok in domain/ packages
  • Domain services via @Bean only (no @Service)
  • JpaEntity in adapter/outbound/persistence/ only
  • @Transactional at adapter layer only
  • Ports named correctly (UseCase/Query/Port)
  • Soft-delete enforced, no hard deletes
  • All list queries scoped to authenticated user
Phase 3B — Security Review
Auth, OWASP & Data Exposure
security-reviewer
  • Auth token checked on every new endpoint
  • Other user's resources return 404, not 403
  • No financial data logged at INFO level
  • Parameterized SQL — no string concatenation
  • No sensitive fields in response DTOs
  • Inactive records excluded from all queries
Phase 3C — Performance Review
Queries, Indexes & Bundle
performance-reviewer
  • No N+1 in new JPA repositories
  • JOIN FETCH / projections for collections
  • DB indexes for new filter/sort columns
  • List endpoints paginated or limited
  • Aggregates use DB-level, not in-memory
  • Frontend bundle size impact reviewed
Phase 3D — Dependency Review NEW
New Library Audit
tech-lead
  • Any new Gradle / npm dependencies justified?
  • License compatible (no GPL in commercial code)?
  • Known CVEs? (check Dependabot / OWASP NVD)
  • Can existing dependencies solve the problem?
Phase 3E — Consolidation & Remediation
Resolve All Review Findings → Final LGTM
tech-leadfull-stack-dev
  • tech-lead consolidates findings from all streams
  • Hard blockers from any stream must be resolved before merge
  • Advisory items addressed or documented as accepted risk
  • Developer fixes → re-requests review on changed files only
GATE: All hard blockers resolved · tech-lead LGTM · PR ready to merge
⟵ PARALLEL ⟶
Phase 4A — Automated Tests
Unit + Integration + Coverage Gates
qa-automation-tester
  • Domain service unit tests (pure Java)
  • Repository tests for JPA adapters
  • Acceptance / integration tests vs real PostgreSQL
  • Every acceptance criterion has a passing test
  • Domain service coverage ≥ 90%
  • Every use case / query: 100% coverage
  • Migration validation on seed data NEW — run Liquibase against a DB with production-like data to catch column/constraint errors
Phase 4B — Accessibility & Visual
WCAG 2.1 AA + Mobile Review
ux-ui-designer
  • WCAG 2.1 AA on all new screens
  • 375px viewport — no overflow, no horizontal scroll
  • Keyboard nav on all interactive elements
  • Color contrast 4.5:1 (normal) / 3:1 (large)
  • Loading / empty / error states implemented
  • shadcn/ui variants correct — no one-off overrides
Phase 4C — QA Gate
All Checks Must Pass
qa-automation-testerux-ui-designer
  • All automated tests pass, no failures
  • Coverage thresholds met (90% domain, 100% use cases)
  • All acceptance criteria verified by a test
  • Accessibility checklist complete
  • Migration validation on seed data passed
GATE: QA gate passed — no exceptions
Phase 5 — Documentation NEW
API Spec, Changelog & Release Notes
full-stack-devtech-lead
  • API documentation — update OpenAPI/Swagger spec for all new or changed endpoints
  • Changelog — add entry in CHANGELOG.md with version, date, and summary of changes
  • Release notes — human-readable summary for stakeholders (what changed and why)
  • ADR update — confirm architecture-decisions.md reflects any decisions made during implementation
  • README update — if any new env vars, setup steps, or commands were added
GATE: API spec updated · changelog entry added · README accurate
Phase 6A — Pre-Deploy Checklist
Deploy Readiness
devops-engineer
  • Liquibase migration backward-compatible or maintenance window planned
  • No breaking API changes without version bump
  • Env vars / secrets updated in target environment
  • Docker images build successfully
  • CI pipeline green on the feature branch
  • PR merged to main
Phase 6B/6C — Deploy + Smoke Test
Release & Immediate Verification
devops-engineerqa-automation-tester
  • Deploy via CI/CD (make start)
  • App starts + health endpoint responds
  • Auth flow works end-to-end
  • Feature verified manually
  • Liquibase migrations applied cleanly
  • No error spike in logs within 5 min
GATE: Smoke test passes
Phase 6D — Monitoring Setup NEW
Observability Verification
devops-engineer
  • Confirm new endpoints appear in request logs
  • Confirm new DB queries are visible in slow query log if needed
  • Set up any new dashboard panels for the feature (error rate, latency)
  • Verify alerts fire correctly for new failure modes
Phase 6E — Stakeholder Confirmation
Close the Feature Loop
product-owner
  • Requester confirms feature works as expected in production
  • Closes loop opened at Phase 1D Stakeholder Gate
  • Any follow-up items logged as new feature requests

HOTFIX Track

Production bug · data integrity · security vulnerability — Accelerated pipeline. Scope is strictly limited to the broken behavior.

Phase 0 — Intake
HOTFIX Classification
engineering-manager
  • Confirm production is broken, data at risk, or security is compromised
  • Scope strictly limited to the broken behavior — no other changes
  • Notify team of incident before any fix is written
GATE: Incident scope confirmed, team notified
Phase 1 — Diagnosis
Root Cause Analysis
tech-lead
  • Identify exact root cause — not just symptoms
  • Determine minimal fix scope
  • Confirm no refactors bundled in
  • Domain zone rules still apply (no waivers)
Phase 2 — Fix
Targeted Implementation
full-stack-dev
  • Apply minimal fix — no scope creep
  • No refactoring surrounding code
  • No new features bundled in
  • ./gradlew test passes after fix
Phase 3 — Architecture Review (3A only)
Domain Zone Check
tech-lead
  • Domain zone checklist — same rules, accelerated pass
  • Security check only if the hotfix touches auth or data exposure
  • No full performance review needed
GATE: Architecture checklist clean
Phase 4 — Targeted Tests (4A only)
Verify the Bug is Fixed
qa-automation-tester
  • Write a failing test that reproduces the bug first
  • Confirm the fix makes that test pass
  • Run full test suite — confirm no regressions
GATE: Bug-reproducing test passes · no regressions
Phase 5 — Accelerated Deploy
Deploy + Incident Documentation
devops-engineer
  • May bypass normal release schedule — team already notified (Phase 0)
  • Smoke test immediately after deploy
  • Incident documented: root cause, fix applied, affected users
  • Follow-up FEATURE or CHORE ticket logged if systemic fix is needed
GATE: Smoke test passes · incident doc written

CHORE Track

Dependency upgrade · refactor · no domain change — Abbreviated pipeline, no analyst or UX phases.

Phase 0 — Intake
CHORE Classification
engineering-manager
  • Confirm: no domain model changes, no new ports, no new endpoints
  • Confirm: no API contract changes
  • If domain model changes needed → reclassify as FEATURE
GATE: Confirmed no domain change
Phase 2 — Implementation
Backend Only (no UX review unless frontend is affected)
full-stack-dev
  • Apply the targeted change (upgrade, refactor, cleanup)
  • No new abstractions beyond what is needed
  • ./gradlew test passes after change
GATE: Build clean · tests pass
⟵ PARALLEL ⟶
Phase 3A — Architecture Review
Verify No Domain Leakage
tech-lead

  • No accidental cross-context imports
  • Liquibase migrations (if any) correct
GATE: No domain violations
Phase 3D — Dependency Review NEW
New Library Audit
tech-lead
  • Upgraded version — any breaking changes?
  • Known CVEs in new version?
  • License still compatible?
  • Bundle size impact if npm package
Phase 4 — Tests (4A only)
Confirm Nothing Broken
qa-automation-tester
  • Run full test suite: ./gradlew test + ./gradlew integrationTest
  • All existing tests must pass — no regressions
  • No new tests required unless the chore fixes a coverage gap
GATE: All existing tests pass
Phase 5 — Normal Deploy
Standard Release
devops-engineer
  • Normal deploy via CI/CD
  • Smoke test: app starts, no error spike
  • No stakeholder confirmation required

SPIKE Track

Technical investigation · proof-of-concept · ADR research — Output is a document, never production code.

Phase 0 — Intake
SPIKE Classification
engineering-manager
  • Confirm: output will be a document or ADR, not production code
  • Define the specific question the spike must answer
  • Set a time-box (do not let spikes run indefinitely)
GATE: Question defined · time-box set
Phase 1B — Architecture Research
Options Analysis & ADR Authorship
tech-lead
  • Research options for the architectural question
  • Evaluate against hexagonal architecture constraints (ADR-014)
  • Consider bounded context boundary impact
  • Prototype if needed — prototype code is never merged to main
  • Author ADR: Context → Options → Decision → Rationale → Consequences
  • Save to .claude/agent-memory/tech-lead/architecture-decisions.md
GATE: ADR complete and reviewed
Output — ADR Document Only
No Code · No Tests · No Deploy
  • ADR filed in architecture-decisions.md
  • Decision and rationale documented for future reference
  • If spike reveals a new feature is needed → start a FEATURE track request
  • Prototype code discarded or kept in a branch marked "spike — do not merge"
